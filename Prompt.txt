Devi implementare un sistema completo di gestione multi-terminale per Remote VibeCoder. Questa è una modifica strutturale che tocca il backend Express, il bridge WebSocket PTY, il routing delle sessioni tmux e l'intera architettura del frontend React. Leggi attentamente il CLAUDE.md prima di procedere e analizza server/pty.js, server/routes/sessions.js e tutti i componenti esistenti in client-src prima di scrivere una sola riga di codice.

AREA 1 - ESTENSIONE DEL BACKEND PER SESSIONI MULTIPLE. Attualmente il sistema gestisce una sessione tmux per repo con naming convention claude-{reponame} e il WebSocket si connette tramite il path /ws/pty/:repo. Devi estendere questo sistema per supportare sessioni tmux multiple e indipendenti. Ogni terminale aperto deve corrispondere a una sessione tmux distinta con un identificatore univoco composto da repo, tipo di sessione e un suffisso incrementale o UUID corto, ad esempio claude-{reponame}-{terminalId}. Aggiorna server/routes/sessions.js per esporre le API necessarie a creare, elencare, rinominare e terminare sessioni multiple associate allo stesso repo o a repo diversi. Aggiorna server/pty.js per gestire il bridge WebSocket verso sessioni tmux identificate dal nuovo schema. Il WebSocket path diventa /ws/pty/:sessionId dove sessionId è l'identificatore univoco della sessione. Ogni sessione deve includere nei metadati: sessionId, repo di appartenenza, working directory attiva, label leggibile, timestamp di creazione. Esponi questi metadati tramite GET /api/sessions che deve restituire la lista completa delle sessioni attive con i relativi metadati.

AREA 2 - OPZIONI DI APERTURA DI UN NUOVO TERMINALE. Nella schermata terminale devi aggiungere un pulsante dedicato all'apertura di nuovi terminali. Alla pressione di questo pulsante si apre un menu o un bottom sheet con le seguenti quattro opzioni.

Opzione 1, Clona terminale: apre una nuova sessione tmux indipendente con working directory identica a quella del terminale corrente. Il comando di spawn deve essere tmux new-session con il flag -c impostato sul path corrente del terminale sorgente, ricavato chiamando tmux display-message -p #{pane_current_path} sulla sessione attiva.

Opzione 2, Apri in sotto-cartella: mostra un file browser ad albero che esplora ricorsivamente il filesystem a partire dalla root del repo corrente usando l'endpoint esistente GET /api/repos/:name/tree. L'utente può navigare aprendo e chiudendo directory, selezionare una cartella di destinazione e confermare con un pulsante dedicato. La nuova sessione tmux viene avviata con -c impostato sul path assoluto della cartella selezionata.

Opzione 3, Apri in progetto esterno: mostra un selettore minimalista delle repository disponibili, recuperato da GET /api/repos, distinto visivamente dalla main page ma funzionalmente equivalente. Se il repo selezionato non è ancora clonato localmente deve essere disponibile un pulsante inline per clonarlo tramite POST /api/repos/clone prima di procedere. Dopo la selezione del repo l'utente può opzionalmente navigare nelle sotto-cartelle del progetto selezionato con lo stesso file browser descritto nell'Opzione 2. La sessione viene aperta nella root del repo esterno o nella sotto-cartella selezionata.

Opzione 4, Terminale libero: apre una sessione tmux shell senza repo associato, con working directory impostata sulla home dell'utente, usando il flag shell=true già supportato da POST /api/sessions/:repo o creando un endpoint dedicato POST /api/sessions/_free.

AREA 3 - LAYOUT MOBILE: UN TERMINALE ALLA VOLTA CON NAVIGAZIONE A SIDEBAR. Su viewport mobile, definito come larghezza inferiore a 768px, deve essere visibile un solo terminale per volta a schermo intero. Per navigare tra i terminali aperti devi implementare una sidebar a scomparsa. La sidebar si attiva premendo un pulsante icona posizionato nella top bar del terminale, nel lato opposto rispetto agli altri controlli esistenti. La sidebar scorre verso l'interno dal bordo dello schermo con un'animazione fluida e mostra la lista delle sessioni attive con label, repo di appartenenza, working directory e un indicatore di attività. Ogni elemento della lista permette di switchare al terminale corrispondente. Ogni elemento ha anche un pulsante di chiusura inline. La sidebar si chiude tappando fuori di essa o ripremendo il pulsante icona. Crea questo componente come TerminalSidebar centralizzato in client-src/components con le props necessarie a ricevere la lista sessioni e i relativi handler.

AREA 4 - LAYOUT DESKTOP: FINESTRE DRAGGABILI E SOVRAPPONIBILI. Su viewport desktop, definito come larghezza pari o superiore a 768px, i terminali devono essere visualizzabili come finestre floating draggabili e sovrapponibili all'interno di un workspace container a schermo intero. Ogni finestra terminale deve avere nella propria title bar tre controlli classici di window management posizionati in alto a destra: minimizza, che riduce la finestra a una barra titolo compatta nel workspace, espandi a schermo intero o ripristina le dimensioni precedenti, e chiudi che termina la sessione tmux tramite DELETE /api/sessions/:sessionId. Le finestre devono essere draggabili tramite la title bar usando una libreria leggera compatibile con React oppure implementando il drag manualmente con pointer events. La gestione dello z-index deve portare in primo piano la finestra su cui si clicca. In modalità desktop la sidebar descritta nell'Area 3 deve essere sostituita da una taskbar orizzontale o verticale che mostra le sessioni minimizzate e permette di ripristinarle. Crea un componente TerminalWindow centralizzato in client-src/components che incapsula titolo, controlli, logica di drag e resize, e che renderizza il terminale xterm.js al proprio interno.

AREA 5 - MODALITÀ MOBILE VS DESKTOP: RILEVAMENTO E ADATTAMENTO DINAMICO. Crea un hook React useMobileLayout in client-src/hooks che usa ResizeObserver o window.matchMedia per rilevare la modalità corrente e aggiornare lo stato in modo reattivo al cambio di dimensione del viewport. Questo hook deve essere la fonte unica di verità per la distinzione mobile vs desktop in tutta l'applicazione. I componenti TerminalWindow, TerminalSidebar e il workspace container devono usare questo hook per adattare rendering e comportamento senza duplicare logica di rilevamento.

AREA 6 - CENTRALIZZAZIONE DELLE ANIMAZIONI. Crea un file client-src/animations/index.ts o equivalente che esporti come costanti o oggetti riutilizzabili tutte le definizioni di animazione usate nell'applicazione: transizioni della sidebar, apertura e chiusura di modal e bottom sheet, animazione di minimizzazione delle finestre, fade di overlay. Usa CSS custom properties o un sistema basato su Framer Motion se già presente nel progetto, altrimenti usa CSS transitions con valori di durata e easing definiti centralmente. Nessun componente deve definire valori di animazione inline o hardcoded.

AREA 7 - COMPONENTI UI DA CENTRALIZZARE IN client-src/components. Oltre a TerminalWindow e TerminalSidebar già descritti, crea i seguenti componenti generalizzati: FileBrowser che riceve un path iniziale e un handler di selezione e gestisce la navigazione ad albero nelle directory, RepoSelector che mostra la lista repo in formato minimalista con supporto al clone inline, TerminalOpenMenu che è il menu o bottom sheet con le quattro opzioni di apertura descritte nell'Area 2, e WindowManager che in modalità desktop gestisce il workspace delle finestre floating con z-index e stato di ogni finestra. Ogni componente deve avere props tipizzate con TypeScript e non deve contenere logica di business specifica del dominio.

Prima di scrivere codice analizza l'intero contenuto di client-src per mappare i componenti esistenti, verifica quali dipendenze npm sono già installate nel package.json di client-src per non introdurre librerie non necessarie, e pianifica le modifiche al backend verificando la compatibilità con il vincolo di 1GB RAM più 2GB swap dell'e2-micro tenendo presente che ogni sessione tmux aggiuntiva ha un costo in memoria.
